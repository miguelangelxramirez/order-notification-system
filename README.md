# Order Notification System

Two Spring Boot microservices connected via Kafka. When an order is created, a notification is automatically generated using a RAG pipeline backed by pgvector and an LLM (AWS Bedrock or a local fake adapter for development).

Built to explore hexagonal architecture, event-driven communication, AI-generated content with guardrails, and distributed observability.

---

## Architecture

```
Postman / curl
      │
      ▼
┌─────────────────────┐          ┌──────────────────────────────┐
│   order-service     │          │     notification-service      │
│   :8081             │          │     :8082                     │
│                     │          │                               │
│  POST /orders       │          │  Kafka listener               │
│  GET  /orders       │  Kafka   │  Embed order data             │
│                     │ ───────► │  Search pgvector (RAG)        │
│  Postgres           │          │  Call LLM (fake / Bedrock)    │
│  schema: orders     │          │  Validate with guardrails     │
└─────────────────────┘          │  Postgres + pgvector          │
                                 │  schema: notifications        │
                                 └──────────────────────────────┘

Prometheus ◄── /actuator/prometheus ◄── both services
Grafana    ◄── Prometheus
Jaeger     ◄── OTel Collector ◄── both services
```

**End-to-end flow:** `POST /orders` → saved to Postgres → event published to Kafka topic `order-events` → notification-service consumes it → embeds the order text → searches similar documents in pgvector → builds a prompt with that context → LLM generates a notification in JSON → guardrails validate the output (channel whitelist, PII redaction, length limit, fallback on failure) → notification saved → available at `GET /notifications`.

## Running locally

**Prerequisites:** Docker Desktop, Java 21, IntelliJ IDEA (or Maven CLI).

**1. Start the infrastructure**
```bash
docker compose up -d
```
Starts Redpanda, PostgreSQL with pgvector, OTel Collector, Prometheus, Grafana and Jaeger.

**2. Start both services from your IDE**

Run `OrderServiceApplication` and `NotificationServiceApplication` as standard Spring Boot apps. No extra configuration needed — both services default to `app.ai.provider=fake`, so no AWS credentials are required to run locally.

**3. Verify**
```bash
curl http://localhost:8081/actuator/health
curl http://localhost:8082/actuator/health
```

## Try it

**Create an order:**
```bash
curl -X POST http://localhost:8081/orders \
  -H "Content-Type: application/json" \
  -d '{"product":"Laptop Gaming","quantity":2,"price":999.99}'
```

## Design notes

- Services communicate asynchronously via Kafka. This means notification-service can be down while orders are created — it will catch up when it comes back.
- notification-service is idempotent: if the same event is delivered twice (Kafka at-least-once), it will not create duplicate notifications. The check is `sourceOrderId + channel`.
- If processing fails after retries, the message goes to `order-events-dlq` instead of blocking the rest of the topic.
- If the LLM returns invalid or unsafe output, guardrails produce a deterministic fallback (`fallback: true`).
- Switching from fake to real Bedrock requires only a config change (`AI_PROVIDER=bedrock`). No code changes. This is the point of the adapter pattern.

## Using AWS Bedrock

Set the following environment variables before starting notification-service:

```bash
AI_PROVIDER=bedrock
AWS_REGION=us-east-1
BEDROCK_TEXT_MODEL_ID=anthropic.claude-3-5-sonnet-20241022-v2:0
BEDROCK_EMBEDDING_MODEL_ID=amazon.titan-embed-text-v2:0
```

Standard AWS credential resolution applies (IAM role, `~/.aws/credentials`, environment variables).
